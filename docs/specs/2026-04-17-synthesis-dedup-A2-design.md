## 问题与共同修复

本文档只描述 A2 与 A1 的**差异点**：描述/职责的合并策略。

## A2 策略：每组一次 LLM 调用重写

对每个 `len(indices) >= 2` 的分组，调用一次 LLM 合成一份统一的描述和职责列表。

### Prompt 形状

**System:**
```
你是技术文档整合专家。以下是从不同来源抽取的同一实体的记录，
请合并成一段统一的描述和一组去重后的职责列表。

要求:
- 描述：一句话概括，不要编造未给出的内容
- 职责：合并语义等价的项，保留所有独立职责

严格输出 JSON:
{"description": "合并后描述", "responsibilities": ["职责1", "职责2"]}
```

**User:**
```
[来源 0] 名称: 应急系统
  描述: 用于本站与中心站网络中断时，由车站自行操作的系统
  职责: 在网络中断时提供应急操作支持 / 控制广播和屏

[来源 108] 名称: 应急系统
  描述: 在主系统故障时切换使用的备用平台，用于控制广播和导向屏
  职责: 控制广播 / 控制导向屏 / 提供应急情况下的客运业务支持
```

### 失败回退

任一环节失败（LLM 超时、JSON 解析失败、字段缺失）都回退到 A1 机械合并，不影响总表生成。

### 单元素组（无重复）

`len(indices) == 1` 时直接沿用原 atom 的 description/responsibilities，不调 LLM。

## 成本预估

xianche 实体 dedup 合并组约 30 个。A2 会为其中 `len >= 2` 的约 20 组各增一次 LLM 调用。

## 取舍

- **成本**：+ ~20 次 LLM 调用（按此系统规模）。
- **确定性**：非确定，重跑文本可能不同。
- **可溯源性**：**降低** — 合成后的描述无法精确对回单条 atom；依靠 prompt 约束"不要编造"降低幻觉。
- **可读性**：预期更干净统一（无重复短语拼接）。

## 验收

与 A1 相同：行数减少、重复项合并、来源聚合。差异由用户通过对比 `synthesis/xianche/实体总表.md` 两版输出判断。

## 范围

同 A1：**只改 entity 渲染**。
