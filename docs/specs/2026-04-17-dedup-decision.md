# 合成去重方案选型决策 — 采用 A1，归档 A2

> 2026-04-17 · 针对 `docs/specs/2026-04-17-synthesis-dedup-A1-design.md` 与 `-A2-design.md` 的选型落地

## 决定

**采用 A1（机械拼接 + 子串去重）。** A2 分支 `fix/dedup-A2-llm-merge` 保留归档，不合并到 main。

## 背景

xianche 实体总表去重失败（同名不同源未合并、LLM 代表项被误删）、`文件来源 / 路径来源` 单一化丢失其他出处。两个候选方案并行实现、运行对比，详见各自的 design 文档和分支。

## 为什么选 A1

`doc-bridge` 架构以**可追溯**为核心（atom + provenance + paragraphs + backlinks + 段落溯源）。去重/合成必须与这条主线一致。

A1 满足这条线：
- 机械拼接每条 description，`/` 分隔的子串**各自可追溯**回一个 atom。
- 完全确定性，重跑同输入得同输出，diff 噪声小。
- 零额外 LLM 调用，零幻觉风险。

## 为什么拒 A2（即使修掉假引用）

A2 在首次采样中输出了形如 `（注：[来源99]描述为...，[来源178]仅提及...）` 的引用。这些数字是 LLM 调用时的 atom **索引**，不出现在最终总表的任何列，也不跨调用稳定——**看似可溯源实则无处可查**。

即使修掉这个假引用（比如把数字换成文件名），A2 仍存在四个结构性问题：

1. **非确定性**：同样 atoms，每次 LLM 调用文本不同。PR diff 噪声、重现 bug 困难。
2. **句级可追溯丢失**：A1 的拼接结构让每段话都能对回一个 atom；A2 把 N 条原始描述揉成一段后，读者看到任一句话**无法判断**它来自哪个 atom 或是 LLM 脑补。
3. **幻觉风险**：prompt 里的"不要编造"LLM 不 100% 遵守。样例 `作为主后台或应急后台的组成模块不能重复开启` 是否普适、是否仅某一来源主张，读者无法验证。
4. **成本 & 延迟**：每次 synthesize 多 ~20 次 LLM 调用，后续系统规模扩大后放大。

对技术文档（本项目为铁路旅服系统资料）而言：**能验证 > 好看**。A1 的啰嗦是**诚实的啰嗦**——它把 N 份文档间的措辞差异摊给读者看；A2 强行合成"官方定义"反而**掩盖**了各源之间的不一致。

## 后续改进路线（如需）

**H1 — 字符串级近似去重**（若 A1 输出太啰嗦再考虑）
- 对 `/` 切分出来的子串做 normalize（空白、标点、全半角），再按 Jaccard 或 前缀/后缀重叠度判近似重复。
- 零 LLM，仍然确定。

**不建议**回到 A2 路线，除非业务明确需要"合成摘要"而**放弃可追溯性**。
