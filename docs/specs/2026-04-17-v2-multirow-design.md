# 总表 V2 布局 — 一源一行

> 2026-04-17 · 在 A1 聚合单行的基础上改为 V2 多行（follow-up to `2026-04-17-dedup-decision.md`）

## 背景

A1 合并到 main 后仍有三个观察到的问题：

1. 同一实体的多份来源用 `/` 拼在一格里，读起来难把"哪段描述/职责/段落对应哪个文件"对上。
2. `术语总表` 段落来源只有 `§1`，没有文件前缀，和 `实体总表` 不一致。
3. `术语总表` 和 `数据流总表` 仍然只显示代表 atom 的单一来源（`_flatten_groups` 丢弃了非代表 atom）——所谓"双向链接未成功"其实是渲染层的锅。

## V2 布局

**一个 atom 一行，名称只在组的第一行出现。**

```
| 实体名称 | 描述 | 职责 | 文件来源 | 路径来源 | 段落来源 |
| 应急系统 | 用于本站与中心站网络中断时…… | 在网络中断时提供应急操作支持 / 控制广播和屏 | 湛江西.docx | …/应急系统.md | 湛江西.docx §1, §7 |
|          | 在主系统故障时切换使用…… | 控制广播 / 控制导向屏 / 提供应急情况下的客运业务支持 | 白云.docx | …/应急系统.md | 白云.docx §40, §41 |
```

**要点**
- **每格内容都能精确对回一个 atom**。A1 的跨源 `/` 拼接被消除，没有"这一格出自哪个 atom" 的模糊地带。
- 名称列在组首行显示，后续行留空——markdown 标准行为，人眼和渲染器都能识别。
- 不需要跨源合并描述/职责，自然没有 A2 的幻觉/非确定性问题。

## 统一改动（三张表都适用）

1. 全部接受 `atoms, groups` 而不是被扁平化的 atoms 列表。
2. 每表按 `for indices in groups: for position, idx in enumerate(indices): 渲染一行` 迭代。
3. 新增 `_atom_source_cells()` 工具函数，统一生成三列（文件/路径/段落）。
4. 段落来源统一格式 `<文件名> §p, §q`。
5. 表头"实体数量/术语数量/数据流数量"仍显示**组数**（语义上的"独立实体数"），行数比组数多。

## 代码清理

删除的死代码：
- `_flatten_groups`（V2 不再需要丢弃非代表 atom）
- `_merge_group_A1`（V2 不再需要合并描述/职责）
- 旧的 `_build_entity_table`（扁平版本，早已被 grouped 版本替代）
- `import asyncio`（A2 残留）

新增的工具：
- `_extract_section(body, heading)` — 替代 glossary 里内联的定义提取逻辑
- `_extract_bullet_list(body, heading)` — 替代旧的 `_extract_responsibilities`
- `_atom_source_cells(ws, atom_path, meta, synthesis_path)` — 三列来源构造

## xianche 验证数据

```

entity:   187 atoms → 141 groups  (与 A1 run 的 133 略有漂移，LLM 非决定性)
```

行数 > 组数（每组一组首 + N 个续行）。`术语总表` 的 `广播接口机` 现在展开为 6 行，清楚显示它在 6 份文档里都有定义——这是本次改动最直接的可见收益。

## 未处理 / 后续

- 同名但被 LLM 分到不同组的残留重复（A1 样例里的 `应急触发后台`、`广播接口机` 2 行各自独立）。后处理"同名组合并"是下一步可加的增量，与 V2 正交。
- 表格太宽在窄屏（移动端/PR diff）不友好。若要改善，可以考虑把 `路径来源` 折叠成"§X"链接而非全文件名，但会损失一点可读性。暂不做。
